# @authently/db

Schema, migrations, RLS policies, and generated types for the Authently
database.

## Layout

```
packages/db/
├── schema.sql           current-state reference (informational; not executed)
├── migrations/          append-only Supabase migration files
│   └── 20260428000001_init.sql
├── scripts/
│   └── generate-types.ts   wrapper around `supabase gen types typescript`
├── src/
│   └── index.ts            re-exports the generated `Database` type
└── types.ts                generated types (committed; regenerated by gen:types)
```

`supabase/migrations/` at the repo root is a **symlink** to
`packages/db/migrations/`. The Supabase CLI reads from `supabase/migrations/`
by convention; the canonical source-of-truth location (per `CLAUDE.md`) is
`packages/db/migrations/`.

## Multi-tenant invariants enforced by the migration

1. **RLS enabled before any data lands.** Each migration runs in a single
   transaction. Tables are created and `enable row level security` is
   applied in the same migration, before any insert is possible.
2. **`workspace_members` policy avoids recursion** by routing the membership
   check through `private.is_workspace_member(uuid)` — a SECURITY DEFINER
   function that bypasses RLS for its own SELECT.
3. **Sign-up trigger is atomic.** `private.handle_new_user()` fires AFTER
   INSERT on `auth.users` and creates `workspaces` + `workspace_members` in
   the same transaction. SECURITY DEFINER + postgres ownership lets it
   bypass RLS for the bootstrap inserts. `search_path` is pinned to `''`
   and every object is fully qualified.

## Common commands

From the repo root:

```sh
pnpm db:migrate       # supabase migration up
pnpm db:reset         # supabase db reset (recreates local DB from migrations)
pnpm db:gen-types     # regenerate packages/db/types.ts from current schema
```

`pnpm db:gen-types` reads from the local Supabase instance by default. Set
`SUPABASE_PROJECT_REF` to target a remote project.

## Adding a migration

Migration filenames are timestamps (`YYYYMMDDHHMMSS_short_name.sql`).
Migrations are append-only — never edit a migration that has shipped.

After adding a migration:
1. Run `pnpm db:reset` locally to verify it applies cleanly from scratch.
2. Run `pnpm db:gen-types` to refresh `types.ts` (commit the result).
3. Optionally regenerate `schema.sql` via `supabase db dump`.

## Slug generation

Workspace slugs are generated as `{kebab-base}-{8-hex-suffix}` where the
base is derived from the user's display name and the suffix is 4 random
bytes. The trigger retries up to 3 times on `unique_violation`. See the
header comment in the init migration for the rationale.
